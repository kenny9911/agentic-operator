## TL;DR

Agentic Operator is **80% of the way to an honest "agent OS" claim** and the harness instincts are correct: every primitive a real platform needs has a place in the schema, the runtime, and the API. What's missing isn't another primitive — it's the **single multi-tenant trust boundary** (real RBAC, V8 isolate, per-tenant key vault) and a small set of operator surfaces that turn raw substrate into a product. The Event Tester is the right probe; ship it, and use it to drive the eval/marketplace flywheel that turns "platform" into a defensible position.

## Event Tester as a probe

**What it proves works.** The spec is ruthlessly L4: it calls existing routes (`POST /v1/events`, `POST /v1/events/:id/replay`), reads from existing tables (`eventTypes`, `events`, `runs`), and adds three sibling endpoints (`/v1/events/catalog`, `/v1/events/stream`, `/v1/events/recent?causality=1`). Acceptance criterion §11 — "a new tenant's `events_v1.json` declares an event with 3 fields; that form, publish, run, recent, causality DAG, audit log all work with zero code change" — is the *exact* litmus test for whether the harness is generic. If shipping Event Tester takes any per-tenant code, the platform thesis is broken. Reading the design and the manifest shape in `models/RAAS-v1/events_v1.json`, it doesn't: the `eventTypes` table (PK = `(tenantId, name)`) is already the universal catalog, and `appendToLedger → inngest.send` is already the universal dispatch. The probe will likely pass on round 2.

**What it stresses.** Three real seams. (1) **SSE dispatch model is unspecified** — the design's per-connection 250 ms SQLite poll doesn't survive multi-process or multi-replica; the AI Architect Round 1 critique calls for an in-process tenant-room `EventEmitter` with a 1 s SQLite catch-up. That same seam is needed by `/v1/stream` and `/v1/runs/:id/logs?follow=1` — fixing it once pays three times. (2) **The `__test` flag isn't actually plumbed through the manifest path** — `register.ts:120-133` inserts `runs` without consulting `event.data.__test`; today's `runs.isTest` is back-filled from the sync code-agent invoke route (`agent-invoke.ts:120-133`). Until the manifest registrar reads `__test`, G5/NFR-7 are aspirational, and the "tests traverse the real harness" principle is *almost but not quite* true. (3) **Tenant scoping must be checked at the query layer for the SSE catch-up**, not the route layer — bare event names like `CLIENT_RULES_PASSED` exist across multiple tenants and the catch-up query must filter by `tenantId`.

**What it unlocks.** Three downstream "agent OS" capabilities that are otherwise blocked: (a) **Evals.** Once `runs.isTest=true` is honestly plumbed and a test publish is one click away, every published event with a known-good expected outcome becomes a reusable eval fixture; the schema-driven form can later grow assertion fields and we get a tiny eval harness for free. (b) **Replay-driven incident triage.** The causality DAG is the visualisation that turns the existing `triggerEventId`/`emittedEventId` join into a debugging tool, dropping MTTR by the PRD's claimed 50%. (c) **Marketplace.** A workflow that "just publishes an event" and "just observes downstream runs" can be exported and re-imported across tenants without any of the per-tenant glue that traditionally kills marketplace plays — the catalog *is* the public API.

## 6-month roadmap to "agent OS"

1. **Sandbox isolation for tenant code (V8 isolate or `worker_thread`)** — *why:* the single biggest gap between "self-host single-org" and "platform". PRD R-7 already flags it; until landed, the OS claim doesn't survive an adversarial tenant. *MVP:* per-tenant `worker_thread` driven by the existing `data/tenants/<slug>/<version>/` loader + a postMessage RPC for tool dispatch; no API contract change. *Leans on:* `packages/runtime/src/tenant-loader.ts`, the existing dynamic-import path, and `agent-sdk` (already process-agnostic).

2. **Real RBAC + platform-admin layer** — *why:* `memberships.role` exists but route guards don't read it; `isPlatformAdmin()` is hard-stubbed `false` so the `?include_system=1` escape hatches are dead code. Honest multi-tenant operation needs *both* per-tenant roles and a real platform-admin role. *MVP:* `requireRole("operator")` Fastify hook reading `memberships.role`; `isPlatformAdmin()` reads from a new `platform_memberships` table; `/v1/admin/*` route group for cross-tenant ops. *Leans on:* `apps/api/src/plugins/auth.ts`, the `memberships` table, the `auditLog` table (writes already happen).

3. **Eval harness on top of Event Tester** — *why:* a platform that runs agents but can't measure them is a framework. Once Event Tester ships, an eval is "publish a captured event + assert a downstream condition." This is the highest-leverage product feature in the roadmap. *MVP:* a new `eval_suites` table referencing `(tenantId, eventName, payloadRef, expectedRunStatus, expectedEmittedEvent)`; a `POST /v1/evals/:id/run` that calls the existing publish path with `__test=true`; a portal view summarising pass/fail. *Leans on:* Event Tester's catalog reader, the existing `runs.isTest` column, the SSE stream for live progress.

4. **Per-tenant BYOK + secret vault** — *why:* the PRD already names this as the v1 → v1.1 trade-off (Q5). Without it, the OS is "self-host with shared keys" — fine for single-customer, fatal for marketplace/SaaS. *MVP:* a `provider_keys` table encrypted with libsodium (master key from `AGENTIC_KMS_KEY`), per-tenant lookup in the gateway before each `chat()`, override of `getProviderKeyEnvOverlay()` in `apps/api/src/services/llm.ts`. *Leans on:* the existing key-overlay shape in `services/llm.ts` and the `auditLog` writes for key rotation.

5. **Agent / tool catalog (marketplace v0)** — *why:* the wedge of "platform" over "framework" is shareable artefacts. *MVP:* `POST /v1/catalog/agents` (publish a manifest agent), `GET /v1/catalog/agents` (browse, no auth required), `POST /v1/catalog/agents/:id/install?tenant=` (clones into the caller's tenant — same atomic flow as `POST /v1/agents`). No payment, no rating; just the import/export discipline that proves the manifest+tool contract is portable. *Leans on:* the existing `POST /v1/agents` deploy path, `workflowVersions.manifestJson` as the wire format, and the audit log.

## Risks to the OS positioning

1. **The Inngest dependency is load-bearing and externally hosted in prod.** Self-host story claims "single VPS" but durable execution implicitly assumes Inngest Cloud or a self-hosted Inngest. If a customer can't or won't deploy Inngest, the OS claim doesn't survive. *Mitigation:* document Inngest as a hard dependency, ship a self-host Inngest in `docker-compose`, or sketch a pluggable durable-executor interface (Temporal, Restate) for v2.

2. **"Platform vs framework" branding tension.** The dual-author wedge (manifest JSON + TypeScript class) is a *framework* idiom (LangGraph, Crew); the runtime ledger + multi-tenant + portal is a *platform* idiom (Vellum, Restack). Externally positioning as "platform" attracts ops buyers; the manifest+code authoring still attracts devs. *Mitigation:* don't choose. Sell "platform" externally, lead with the deploy/observe/govern surfaces (Event Tester, run timeline, audit, rollback), keep the code surface as the "no lock-in" trump card.

3. **The SPA is still Babel-standalone at runtime.** Until Phase 2 lands, the operator surface is unbuildable, unbundleable, unauditable for security review, and unbrandable. This is the single most visible "looks like a prototype" tell.

4. **Schema drift between `packages/runtime` (manifest) and `packages/agent-runtime` (code).** The two-process unification works because both write to the same `runs`/`steps` tables, but the manifest path and the code path each have their own `runs` insert; today they already diverge on `isTest` plumbing. *Mitigation:* extract a single `createRun()` helper in `@agentic/db` that both paths must use.

5. **No metering = no monetization.** Cost cap + token-attribution is in place, but there's no per-tenant invoicing, no overage hooks, no "this agent cost X tokens last month" UI. The OS claim doesn't *require* monetization, but the platform positioning does.

## Final verdict

**Ship.** The substrate is honest, the layering is correct, the tests are real (37 vitest files covering tc-1..tc-61, including end-to-end multi-tenant isolation), and the harness instincts are right. Event Tester is the right next ship — it's the lowest-cost, highest-signal demonstration that the harness is generic and the OS claim holds. After it lands, the priorities are sandbox isolation, real RBAC, and an eval harness on top of Event Tester. With those three, "Agent OS" stops being a positioning bet and starts being a defensible engineering claim.
